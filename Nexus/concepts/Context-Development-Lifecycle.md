# Context Development Lifecycle

## Definition
An SDLC for context assets: created, tested, approved, deployed, retired. Context can no longer live as scattered prompts and isolated team decisions; it becomes a first-class asset that is reusable, reproducible, versioned, and governable. AI handles building/testing/reviewing; humans own approval and deployment.

## Why It Matters
Software engineering created the SDLC to manage how code is built, reviewed, versioned, and deployed. Context needs the same discipline. Without it, context decays into unverified prompts, drifted definitions, and competing instructions.

## Key Ideas
- Change propagation is crucial: redefining core ICP is a structural change, not a small edit
- When a positioning doc (CMO-approved skill) feeds social media, SDR pitch, and analyst call skills, update propagation needs governance
- Three propagation modes: automatic, queued for review by downstream owner, or old version runs until certified
- These are organizational design questions, not prompt engineering questions
- The difference between a layer that compounds and one that contradicts itself

## Tradeoffs
- Automatic propagation (fast but risky) vs queued review (safe but slow) vs manual certification (accurate but expensive)
- Centralized governance vs distributed team ownership
- Version granularity -- how fine-grained should context versioning be?
