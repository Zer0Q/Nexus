# Skills as Primitives

## Definition
Skills are the new primitive for procedural knowledge -- what functions were to software logic. A skill is a reusable, versionable, testable unit of how-to knowledge with its own triggers, edge cases, and owner. When organizations treat skills as first-class assets, procedural knowledge compounds instead of being re-derived.

## Why It Matters
Before software had functions, logic lived as instructions re-derived every time. The function turned logic into something nameable, versionable, testable, and callable from anywhere. Skills do the same for enterprise procedural knowledge -- the way you run the monthly close, qualify a lead, or handle a refund.

## Key Ideas
- Procedural knowledge in enterprises lives as scattered prompts, tribal memory, and improvised steps
- A skill makes that knowledge durable: triggers, edge cases, owner, version history
- Two problems: building the library (mining problem) and maintaining it as business changes (lifecycle problem)
- Skills enable compounding -- the same procedure isn't re-derived by every agent and every employee
- Change propagation is crucial: when a core skill updates, downstream skills need review/propagation
- Skills encapsulate judgment, not just steps — they capture edge cases, quality bars, and heuristics of top performers (Hiten Shah)
- Private skills are the real competitive advantage — marketplace skills are generic starting points; company-specific skills encode methods competitors cannot download
- Evolution pattern: connectors gave access to data, skills add judgment and process, next-gen plugins combine both
- The executor changed: agents can now load playbooks, use tools, inspect files, and execute — the playbook becomes active

## Tradeoffs
- Skill granularity -- too fine = maintenance overhead; too coarse = limited reusability
- Automated skill mining vs human-curated skills
- Centralized skill library vs team-owned skill subsets
